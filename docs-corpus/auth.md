# Authentication and session entries

## A01: password storage and login

**Applies to:** security guidance **`version-unverified`** as a living policy source; Node **18.19.0** KDF API is version-evidenced in [N06](node.md#n06-password-kdf-primitive). **Sources:** [OWASP Password Storage Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html) recommends a slow, salted password hash (Argon2id preferred, scrypt when unavailable; bcrypt for legacy) and rejects fast general hashes; [Node 18.19 `crypto.scrypt`](https://nodejs.org/download/release/v18.19.0/docs/api/crypto.html#cryptoscryptpassword-salt-keylen-options-callback) specifies an asynchronous memory-hard primitive and random salt guidance. **Recency:** living OWASP page and frozen Node release docs inspected 2026-07-24.

Hash the submitted password server-side with an approved adaptive password-hashing library/policy, unique per-password salt, calibrated work factor, and constant-time verification through the library. Store only the encoded hash/parameters; never return it. The repository trusts an input `passwordHash`, returns the created user including it, and login checks only email without comparing a password ([C03](code-audit.md#c03-password-lifecycle-is-not-implemented)). This is a substantive security discrepancy, not a library API detail. Choose exact parameters and migration policy with the internal security owner. [N06](node.md#n06-password-kdf-primitive) links back.

## A02: JWT verification and expiry

**Applies to:** `jsonwebtoken` **9.0.2**. **Evidence:** [upstream README at tag v9.0.2](https://github.com/auth0/node-jsonwebtoken/tree/v9.0.2) documents `jwt.verify` signature and expiration validation, allowed `algorithms`, `TokenExpiredError`, and warns that `jwt.decode` does not verify a signature and should not be used for untrusted messages. **Recency:** immutable tag matching the lockfile; inspected 2026-07-24.

Authenticate with `jwt.verify(token, configuredSecretOrPublicKey, { algorithms, issuer, audience, ... })`, catch verification/expiry errors and return 401, and treat verified claims as untrusted input to validate. Do not set `ignoreExpiration`. `decode` is only inspection, not authentication. [`auth.js`](https://github.com/sahidempiricinfotech-dotcom/mern-docs-assistant/blob/e508acf871cc5b8c09d4928ddfd246c8ecaccc6e/server/src/auth.js#L11) uses `decode` and accepts any payload with `sub`, so forged and expired tokens can pass ([C01](code-audit.md#c01-jwt-decode-is-used-as-authentication)). The signing path also reads `user.id` although the native login record provides `_id` ([C02](code-audit.md#c02-token-subject-field-mismatch)). A hard-coded `dev-secret` fallback must not be used in production. The Express error boundary is [E01](express.md#e01-async-errors-in-express-4-and-5).

## A03: browser token storage

**Applies to:** **`version-unverified` living browser/security guidance**, not a React API. **Sources:** [OWASP HTML5 Security](https://cheatsheetseries.owasp.org/cheatsheets/HTML5_Security_Cheat_Sheet.html#storage-apis) warns against session identifiers in localStorage because JavaScript/XSS can access them; [OWASP Session Management](https://cheatsheetseries.owasp.org/cheatsheets/Session_Management_Cheat_Sheet.html) explains `Secure`, `HttpOnly`, and `SameSite` and the residual CSRF/XSS boundary. **Recency:** living pages inspected 2026-07-24.

Prefer an opaque server-side session or carefully designed `HttpOnly; Secure; SameSite` cookie/BFF for browser credentials over persistent localStorage tokens. `HttpOnly` reduces JavaScript token theft but does not stop a browser from sending a cookie on an attacker-induced request, so use CSRF and origin protections appropriate to the flow. Cross-origin cookie use also requires the paired CORS/client configuration [E04](express.md#e04-credentialed-cors-server-side) and [R05](react.md#r05-credentialed-browser-fetch-and-cors). This is a **Medium** bounded recommendation because the deployment and threat model are not in the repo; it is not an assertion that a cookie alone solves auth.

## A04: refresh-token rotation

**Applies to:** RFC **9700 / BCP 240**, published January 2025; application design remains **Low confidence**. **Evidence:** [RFC 9700 §4.14.2](https://www.rfc-editor.org/rfc/rfc9700.html#section-4.14.2) requires replay detection for public clients through sender-constrained refresh tokens or rotation, describes issuing a new token on each refresh, invalidating the prior token, retaining the relationship, and revoking the active token when replay is detected. The [tagged jsonwebtoken README](https://github.com/auth0/node-jsonwebtoken/tree/v9.0.2) explicitly does not provide an automatic refresh design. **Recency:** immutable January 2025 standard and package tag, inspected 2026-07-24.

The bounded pattern is a server-side token family/grant record, atomic one-time exchange, hashed token storage, replay/family revocation, expiry and logout/password-change revocation, and concurrency policy. But this repository has no refresh endpoint, store, client classification, threat model, revocation policy, or internal runbook. **Low / G-002:** do not prescribe a single production design or grace window from the RFC alone.

## A05: signed session cookie and Mongo store

**Applies to:** **`version-unverified`** package pairing for this app. **Sources and recency:** living [`express-session`](https://github.com/expressjs/session) and [`connect-mongo`](https://github.com/jdesboeufs/connect-mongo) READMEs plus [OWASP Session Management](https://cheatsheetseries.owasp.org/cheatsheets/Session_Management_Cheat_Sheet.html), inspected 2026-07-24. The repo has no pinned session dependencies or proxy/cookie policy; see [V01](versions.md#v01-repository-pinned-stack).

The cookie should hold a signed opaque session ID, while session state lives server-side. Avoid production `MemoryStore`; use a tested compatible Mongo store, strong secret rotation, secure cookie attributes, proxy trust, TTL/touch semantics, regeneration on privilege change, and CSRF controls. This is **Low / G-004** until versions and deployment policy are selected. The framework boundary [E05](express.md#e05-session-middleware-boundary) and store boundary [D06](data-access.md#d06-session-store-compatibility) link both ways.
